## Контекст и охват

Эта заметка детализирует этап 2 процесса онбординга ноды — установку mTLS-канала между дашбордом и агентом. Согласно «Онбордингу ноды в дашборд», к моменту начала этого этапа SSH уже закрыт навсегда, root-пароль стёрт, агент установлен и слушает HTTP без TLS. Канал, защищённый mTLS, который появится в результате этого этапа, остаётся штатным механизмом аутентификации до конца жизни ноды в системе — до `unpair` или полной зачистки, описанных в Удаление_ноды_из_дашборда.md. Это единственный механизм аутентификации между дашбордом и агентом во всей последующей эксплуатации (см. Принцип_работы_агента_ноды.md — сам агент не содержит прикладной логики авторизации, он полагается на то, что TLS-сессия уже провалидирована).

Главная сложность этапа: на его начало не существует ни одного защищённого канала между дашбордом и нодой, а инвариант CORE.md («только дашборд инициирует соединение») должен выполняться и здесь.

### Цепочка доверия

Документ, по сути, описывает не столько обмен HTTP-запросами, сколько непрерывную передачу доверия от одного механизма аутентификации к другому. На этапе 1 доверие устанавливается посредством SSH: подлинность ноды подтверждается проверкой host key (или сознательным отказом от неё через TOFU), а право дашборда управлять нодой — успешной аутентификацией по root-паролю. Это доверие переносится на временный HTTP-канал через одноразовый bootstrap-токен. После выпуска и подтверждённой установки сертификата bootstrap-токен уничтожается, а единственным штатным носителем доверия становится mTLS — до `unpair` или полной зачистки ноды.

Каждый из разделов ниже детализирует одно звено этой цепочки.

## Ограничения

- Соединение всегда инициирует дашборд — в том числе на этом этапе, до появления постоянного mTLS-канала. Агент никогда не открывает соединение к дашборду сам.
- На голом HTTP (без TLS) до установки сертификата у агента нет средств подтвердить дашборду, что он — та самая нода, которая была установлена на этапе 1, а не кто-то, перехвативший трафик или подменивший ответ по этому IP. Этот разрыв должен быть закрыт bootstrap-токеном, переданным через единственный аутентифицированный канал, который у системы есть — SSH-сессию этапа 1.
- Bootstrap-токен — одноразовый и существует только в окне между концом этапа 1 и успешным завершением этапа 2. В v1 это 32 random bytes из CSPRNG, закодированные как base64url; эффективная энтропия — 256 bit. TTL токена — 15 минут, абсолютный максимум bootstrap-окна — 30 минут. Он не предназначен для долгосрочной аутентификации (эту роль берёт на себя mTLS) и не подменяет собой root-пароль как секрет — у него отдельное назначение и отдельный жизненный цикл.
- После успешной установки mTLS bootstrap-токен уничтожается с обеих сторон: дашборд инвалидирует его в БД, агент удаляет его из своего хранения. Raw token не логируется и не хранится в SQLite: дашборд держит его только в памяти текущего onboarding-процесса, а в БД хранит status, expiry и hash токена. Агент также хранит hash токена и expiry, а не raw token.
- Временный HTTP-эндпоинт для обмена CSR/сертификатом существует только до установки mTLS. После этого момента он либо отключается агентом, либо перестаёт быть осмысленным каналом — вся последующая коммуникация идёт поверх mTLS через эндпоинты `/v1/scripts/*` и административные endpoint'ы агента.
- **mTLS-проверка дополняется прикладной проверкой через БД дашборда.** Сертификат агента считается принадлежащим управляемой ноде только если после успешного TLS-handshake его fingerprint/serial/public key fingerprint сопоставлен с активной записью `node` в локальной БД. Удаление `node` удаляет это прикладное доверие без отдельного CRL.
- **Завершение доверия выполняется через `unpair`.** При штатном online-удалении ноды дашборд просит агента удалить trust anchor/CA дашборда, сертификат агента, private key и локальное pairing-состояние. После `unpair` старый дашборд больше не должен быть авторизован для штатных команд на этой VPS.

## Обзор

### Проверка host key при SSH-подключении (этап 1)

Поскольку нода — произвольный VPS у произвольного хостера, у дашборда нет программного источника эталонного host key (нет единого API, общего для всех хостеров, через который его можно было бы получить заранее). Источником эталона может быть только сам пользователь.

Перед установкой пользователь может получить fingerprint host key по каналу, независимому от SSH (например, через VNC/веб-консоль хостера, IPMI, KVM-over-IP или любую другую внеполосную консоль, которую предоставляет конкретный хостер), и вручную ввести его в дашборд. Формулировка намеренно не привязана к конкретному провайдеру или способу доступа — важно только, что канал получения fingerprint независим от сетевого пути, по которому пойдёт сам SSH.

Исходя из этого:

- В форме добавления ноды, помимо IP и root-пароля, есть **опциональное** поле для host key fingerprint.
- **Fingerprint указан** — дашборд требует строгого совпадения при SSH-подключении. При несовпадении подключение обрывается до ввода пароля, нода не добавляется, пользователю показывается явная ошибка о возможной подмене сервера.
- **Fingerprint не указан** — дашборд работает по TOFU: принимает host key, присланный сервером при первом подключении, и сохраняет его в БД как доверенный для этой ноды на будущее (любое последующее изменение key для уже добавленной ноды — повод для тревоги, а не молчаливого переподключения). Пользователю явно показывается предупреждение, что проверка подлинности хоста была пропущена и есть риск MITM именно в момент этого первого подключения.

Так компромисс делается явным и видимым для пользователя, а не скрытым дефолтом: можно получить полную защиту ценой одного дополнительного действия, а можно сознательно пропустить его, понимая риск. Fingerprint остаётся опциональным входным параметром, а не обязательным — обязательность противоречила бы самой идее добровольного выбора уровня защиты.

**Модель угроз.** Проверка host key защищает исключительно от MITM-атаки на сетевом пути между дашбордом и нодой в момент первичного SSH-подключения — то есть от ситуации, когда кто-то на этом пути (скомпрометированный промежуточный узел, спуфинг в локальной сети и т. п.) подставляет себя вместо настоящей ноды. Она **не защищает** ни от компрометации самой ноды, ни от компрометации машины, на которой работает дашборд — это другие векторы атаки, вне области действия этого механизма.

Этот выбор напрямую защищает и весь bootstrap-токен (см. ниже) — если host key не проверен и SSH-сессия перехвачена, токен, переданный по ней, скомпрометирован вместе с root-паролем, и вся последующая схема mTLS-bootstrap теряет смысл.

### Передача bootstrap-токена (продолжение этапа 1)

В конце установочного скрипта, который дашборд выполняет по SSH на этапе 1, дашборд генерирует случайный одноразовый bootstrap-токен: 32 random bytes из CSPRNG, base64url-представление для передачи в конфиг агента. Токен передаётся агенту как часть установки и одновременно связывается с этой нодой на стороне дашборда. Raw token дашборд держит только в памяти текущего onboarding-процесса; в БД сохраняются `bootstrap_status`, `bootstrap_expires_at` и hash токена. Агент тоже хранит hash токена и expiry, а не raw token. Это происходит ещё внутри SSH-сессии, до того как канал закрывается навсегда — то есть токен передаётся по соединению, подлинность которого подтверждена проверкой host key (или TOFU, если пользователь сознательно отказался от проверки), а доступ к серверу подтверждён успешной аутентификацией по root-паролю.

### Опрос агента дашбордом (bootstrap-аутентификация)

Сразу после завершения этапа 1 дашборд начинает опрашивать агента по HTTP — это инициатива дашборда, инвариант соблюдён. У агента появляется временный bootstrap-эндпоинт, существующий исключительно до установки mTLS и принимающий только операции, необходимые для выпуска сертификата — он не пересекается со штатным API скриптов из Принцип_работы_агента_ноды.md.

Каждый такой запрос дашборд сопровождает bootstrap-токеном (например, заголовок `Authorization: Bootstrap <token>`). Агент проверяет токен против того, что у него сохранён в конфиге с этапа 1, и только при совпадении обрабатывает запрос. Без токена или с неверным токеном — запрос отклоняется.

Это закрывает основной риск этапа: тот, кто способен отвечать на HTTP-запросы дашборда на этом этапе (например, заняв тот же IP, или вклинившись в сетевой путь к ноде), но не присутствовал на SSH-сессии этапа 1 и поэтому не знает значения токена, не может ни выдать себя за агента и подменить CSR, ни заставить дашборд подписать сертификат для чужого ключа. Важно подчеркнуть: bootstrap-токен не усиливает безопасность самого SSH-подключения и не является дополнительным фактором аутентификации в привычном смысле — его единственная задача состоит в переносе доверия, уже установленного во время SSH-сессии, на временный HTTP-канал до момента появления mTLS.

### Генерация CSR и выпуск сертификата

1. Агент генерирует ключевую пару и CSR (если ещё не сгенерировал — генерация может занять время, поэтому первые опросы дашборда могут получать ответ «ещё не готово»).
2. Дашборд получает CSR в ответ на опрос, авторизованный bootstrap-токеном.
3. Дашборд подписывает сертификат и отправляет его агенту — также инициируя этот запрос сам (например, в рамках того же цикла опроса: дашборд присылает подписанный сертификат как часть следующего HTTP-обмена).
4. Агент устанавливает полученный сертификат и пытается перезапустить HTTP-сервер уже с mTLS. Если перезапуск не удался — bootstrap-токен **не** уничтожается: агент остаётся доступным по bootstrap-эндпоинту, чтобы дашборд мог повторить выпуск сертификата при следующем опросе.
5. Только после того как перезапуск прошёл успешно и дашборд подтвердил работоспособность канала пробным запросом поверх mTLS, обе стороны уничтожают bootstrap-токен — дашборд инвалидирует его в БД, агент удаляет его из своего хранения.

Канал дашборд↔агент с этого момента защищён mTLS и остаётся штатным каналом до `unpair` или полной зачистки ноды. Дальнейшая коммуникация (в том числе заливка метрик-скриптов на этапе 3) идёт исключительно по этому каналу.

### Прикладная валидация сертификата агента

Успешный mTLS-handshake доказывает только криптографическую корректность сертификата: он подписан доверенным CA, не истёк и соответствует предъявленному private key. Для дашборда этого недостаточно, чтобы считать endpoint активной управляемой нодой.

После TLS-проверки дашборд дополнительно сверяет идентичность peer certificate с локальной БД: certificate fingerprint, public key fingerprint или serial должен быть привязан к активной записи `node`. Если такой записи нет, запрос к агенту не считается обращением к управляемой ноде, даже если сертификат криптографически валиден.

Это правило используется вместо CRL в v1. Удаление записи `node` делает сертификат агента непригодным для дашборда на прикладном уровне. Обратная сторона доверия — чтобы сам агент перестал доверять старому дашборду — закрывается операцией `unpair`, описанной в Удаление_ноды_из_дашборда.md.

## Детальный дизайн

**Алгоритм host key.** Дашборд жёстко ограничивает список допустимых алгоритмов host key на стороне SSH-клиента (asyncssh) одним значением — `ed25519`. Это снимает риск рассинхрона между тем, что ввёл пользователь, и тем, что согласует SSH-клиент при algorithm negotiation: раз клиент сам диктует серверу единственный приемлемый алгоритм, сверка всегда происходит с тем же ключом, fingerprint которого получил пользователь. В форме ввода это явно отражено — поле подписано как «fingerprint ed25519-ключа», а не обезличенно как «host key fingerprint». Современные дистрибутивы (Debian/Ubuntu) генерируют ed25519-ключ по умолчанию; если у конкретной ноды такого ключа нет, подключение завершится понятной ошибкой ещё до попытки сверки fingerprint. Поддержка нод без ed25519-ключа (например, очень старых дистрибутивов с устаревшим OpenSSH) сознательно не является целью проекта.

### Bootstrap-токен и окно ожидания

Параметры v1:

```text
bootstrap_token_bytes = 32
bootstrap_token_encoding = base64url
bootstrap_token_entropy = 256 bit
bootstrap_token_ttl = 15 minutes
bootstrap_absolute_window = 30 minutes
poll_interval_initial = 2 seconds
poll_interval_after_60s = 5 seconds
```

Дашборд начинает polling сразу после закрытия SSH-сессии этапа 1. Первую минуту он опрашивает агента каждые 2 секунды, затем — каждые 5 секунд до истечения TTL. Если за 15 минут CSR не получен и mTLS не установлен, onboarding переходит в состояние `failed_bootstrap_timeout`. Абсолютный максимум bootstrap-окна — 30 минут: даже если локальный процесс дашборда завис или UI не обновился, агент не должен принимать bootstrap-токен дольше этого окна.

Raw token не персистится в SQLite. Если дашборд падает между этапом 1 и этапом 2, автоматическое продолжение bootstrap в v1 не поддерживается: пользователь запускает onboarding/repair-flow заново, SSH-этап перезаписывает старый bootstrap-token новым, старый token истекает или заменяется.

### Bootstrap endpoint'ы

Bootstrap API живёт в отдельном namespace и не пересекается со штатным `/v1/scripts/*` API:

```text
GET  /bootstrap/v1/status
GET  /bootstrap/v1/csr
POST /bootstrap/v1/certificate
```

Все запросы требуют заголовок:

```http
Authorization: Bootstrap <token>
```

Минимальные ответы:

```text
202 csr_not_ready              — агент ещё генерирует ключ/CSR
200 csr_ready                  — CSR готов и возвращён дашборду
200 certificate_installed      — сертификат установлен, агент перезапускает штатный mTLS endpoint
401 invalid_bootstrap_token    — токен отсутствует, неверен или истёк
410 bootstrap_closed           — агент уже paired/unpaired, bootstrap-режим закрыт
```

После успешной установки сертификата дашборд делает пробный запрос поверх mTLS. Только после подтверждения обе стороны уничтожают bootstrap-token/state: дашборд помечает bootstrap как `paired`, агент отключает bootstrap endpoint.

### Повторный SSH к уже онбордженной ноде

Штатный lifecycle после этапа 1 не использует SSH повторно. Повторный SSH считается repair/debug-сценарием, а не обычным способом управления нодой.

Если такой сценарий всё же запускается, дашборд обязан применить сохранённый host key policy:

- если для ноды сохранён ed25519 fingerprint, новое SSH-подключение требует строгого совпадения;
- если fingerprint не совпал, подключение обрывается до ввода пароля;
- дашборд показывает пользователю ожидаемый fingerprint, полученный fingerprint, тип ключа и IP/host, к которому пытался подключиться;
- автоматическое обновление сохранённого host key запрещено.

Несовпадение host key для уже известной ноды считается security warning, а не обычной ошибкой соединения. Пользователь может продолжить только через отдельный manual repair-flow, который должен явно зафиксировать смену host key как осознанное действие.

### Диагностика fingerprint mismatch

При несовпадении fingerprint безопасно показывать:

```text
expected_fingerprint
received_fingerprint
key_algorithm = ed25519
node_ip
```

Эти значения не являются секретами и помогают отличить опечатку, неправильную VPS и возможный MITM. Root-пароль, bootstrap-token и private keys в диагностике не показываются и не логируются.


## Открытые вопросы

0. ~~**Нужен ли CRL для удалённых нод.**~~ **Закрыто для v1:** CRL не используется. Дашборд проверяет сертификат агента через активную запись `node` в БД, а штатное online-удаление выполняет `unpair` на стороне агента. CRL не решает задачу физической зачистки VPS и считается избыточным для локальной single-dashboard модели.
1. ~~**Формат и длительность жизни bootstrap-токена.**~~ **Закрыто для v1:** 32 random bytes из CSPRNG, base64url, 256 bit entropy; TTL — 15 минут, абсолютное bootstrap-окно — 30 минут. Raw token хранится только в памяти текущего onboarding-процесса, в БД — только hash/status/expiry.
2. ~~**Поведение при повторном опросе после уже выпущенного сертификата.**~~ **Закрыто для v1:** bootstrap endpoint после успешного pairing отключается и отвечает `410 bootstrap_closed` либо становится недоступен; штатная коммуникация идёт только через mTLS. Автоматическое восстановление bootstrap после падения дашборда между этапами 1 и 2 не поддерживается — пользователь повторяет onboarding/repair-flow.
3. ~~**Окно ожидания при опросе.**~~ **Закрыто для v1:** polling каждые 2 секунды первую минуту, затем каждые 5 секунд до TTL 15 минут. После TTL onboarding получает `failed_bootstrap_timeout`.
4. ~~**Поведение при попытке повторного SSH к уже онбордженной ноде.**~~ **Закрыто для v1:** повторный SSH не является штатным путём и допускается только как repair/debug-flow. Если он всё же выполняется, дашборд требует совпадения сохранённого ed25519 host key fingerprint; при несовпадении подключение обрывается до ввода пароля, автоматическое обновление fingerprint запрещено.
5. ~~**Диагностика при несовпадении fingerprint.**~~ **Закрыто для v1:** UI может показать ожидаемый fingerprint, полученный fingerprint, алгоритм ключа (`ed25519`) и адрес ноды. Эти данные не являются секретами; root-пароль, bootstrap-token и private keys не показываются и не логируются.
