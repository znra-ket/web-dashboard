## Контекст и охват

Эта заметка детализирует подсистему агента, отвечающую за исполнение скриптов на ноде. Согласно CORE.md, агент принимает команды только от дашборда (соединение всегда дашборд → агент, NAT-инвариант), исполняет их локально и возвращает результат синхронно, в том же HTTP-обмене.

Единый механизм для всех скриптов: агент хранит скрипты по хэшу содержимого и умеет их загружать, запускать и удалять по хэшу. Никакого деления на "системные" и "пользовательские" скрипты нет. Скрипты для сбора метрик (xray_status, detect_stack, speedtest) — это обычные строки в таблице скриптов дашборда, просто созданные программно при инициализации дашборда, а не руками через UI. На ноду они попадают тем же путём, что и любой другой скрипт.

## Ограничения

- Соединение всегда инициирует дашборд. Агент никогда не открывает соединение к дашборду сам — ни для доставки результата, ни для чего-либо ещё.
- Транспорт — mTLS (отдельная заметка). Шифрование и аутентификация — свойство всей TLS-сессии целиком, поэтому ответ (response) защищён не слабее запроса — это не два отдельных события, а один и тот же провалидированный канал. На стороне дашборда TLS-проверка дополняется прикладной проверкой: сертификат агента должен быть сопоставлен с активной записью `node` в локальной БД.
- Агент может быть штатно отвязан от дашборда через административную операцию `unpair`: агент удаляет локальное доверие к старому дашборду и перестаёт принимать от него штатные команды. Полная зачистка ноды включает `unpair`, но дополнительно удаляет файлы агента, скрипты и служебные директории.
- Выполнение скрипта — синхронное: HTTP-запрос на выполнение блокируется до завершения скрипта или истечения таймаута. Стриминг и job+polling сознательно не используются в v1.
- Агент ничего не знает про "имена" скриптов (`speedtest`, `xray_status` и т.п.) — он оперирует только хэшами содержимого. Человекочитаемые имена и их соответствие текущим хэшам — это ответственность дашборда, не агента.
- Единственный источник истины для хэша скрипта — сам агент. Если клиент (дашборд) передаст в `upload` какое-либо поле вида `hash` — оно полностью игнорируется. Хэш всегда вычисляется агентом от фактически полученных байт.
- Директория хранения скриптов (по хэшам) доступна на запись только процессу агента — никакой другой пользователь системы не может предварительно создать там файл или symlink, что снимает риски атак через заранее подложенные symlink при `rename()`.
- **Лимиты v1 фиксированы, а не являются открытой настройкой.** Агент ограничивает upload, execute body, stdout/stderr, timeout, concurrency, request_id-cache, число процессов, open files, память и workdir. Превышение лимита — явная ошибка (`413`, `422`, `429` или failed-result), а не молчаливое выполнение без ограничений.

## Обзор

Endpoint'ы на агенте:

```
POST   /v1/scripts/upload       — заливка кода скрипта, идентификатор — хэш содержимого
POST   /v1/scripts/execute      — запуск ранее залитого скрипта по хэшу
DELETE /v1/scripts/{hash}       — удаление скрипта по хэшу
GET    /v1/info                 — версия агента и поддерживаемые возможности

POST   /v1/admin/unpair         — разрыв доверия агента к текущему дашборду (точное имя endpoint'а уточняется)
POST   /v1/admin/uninstall      — полная зачистка компонентов web-xray-dashboard на VPS (точное имя endpoint'а уточняется)

(bootstrap, временный)          — `/bootstrap/v1/status`, `/bootstrap/v1/csr`, `/bootstrap/v1/certificate` на этапе онбординга (см. Установка_mtls_соединения_дашборд_-_агент.md)
```

**Список выше, за исключением bootstrap-строки, — это API агента в штатном режиме, поверх уже установленного mTLS.** Административные endpoint'ы `unpair`/`uninstall` не являются частью механики исполнения скриптов и не используют `request_id`-контракт `execute`: это отдельные lifecycle-операции ноды. Bootstrap-эндпоинт существует только до установки mTLS (этап 2 онбординга, см. Онбординг_ноды_в_дашборд.md), защищён отдельной bootstrap-токен-аутентификацией вместо mTLS и не пересекается по контракту со штатными `/v1/scripts/*` (идемпотентность, `request_id`, лимиты и т.д. ниже описаны только для них). Это отдельная, временная поверхность на агенте с собственной моделью угроз — конкретика по-прежнему в mTLS-заметке, здесь он указан только чтобы список выше не читался как исчерпывающий.

Скрипты на ноде не одноразовые — они кешируются и переиспользуются между запусками без повторной заливки кода.

### Оптимизированный флоу запуска

Чтобы не делать лишний round-trip на каждый запуск уже знакомого агенту скрипта, `execute` сразу принимает хэш и пытается выполнить:

1. Дашборд шлёт `execute` с хэшем нужного скрипта.
2. Если агент находит скрипт с таким хэшем у себя — выполняет сразу и отдаёт результат в response. Это основной, самый частый путь.
3. Если не находит — отвечает `404` (скрипт неизвестен агенту).
4. Только в этом случае дашборд делает `upload` (заливает код), получает подтверждение, и повторяет `execute`.

То есть отдельный шаг "проверка наличия" не нужен как отдельный endpoint — его роль играет обычный 404 на execute, который запускает upload только когда он действительно нужен (первый запуск скрипта на этой ноде, либо скрипт обновился и хэш изменился).

### Флоу обновления скрипта

Когда у скрипта в таблице дашборда меняется хэш (был A, стал B), дашборд сначала меняет desired-state в своей БД (`script.current_hash = B`), а затем приводит физическое состояние активных нод к новому состоянию best-effort:

1. Дашборд выбирает все ноды с активной привязкой к этому `script_id` через `node_script`. Это не история фактических запусков, а текущее декларативное намерение «скрипт должен быть доступен на этой ноде».
2. Для online-нод дашборд заранее выполняет `upload` нового hash B. Для offline-нод отдельная очередь upload не нужна: при следующем `execute` обычный путь `execute → 404 → upload → execute` доставит B лениво.
3. После изменения БД дашборд пересчитывает `desired_hashes(node)` для каждой затронутой ноды. Если старый hash A больше не нужен конкретной ноде ни через один активный `script`, создаётся `node_hash_gc(node, A)`.
4. Для online-ноды GC выполняется сразу через `DELETE /v1/scripts/A`; для offline-ноды остаётся pending до следующего успешного контакта.

Физическое удаление старого hash никогда не делается вслепую «на тех же нодах»: один и тот же content hash может использоваться несколькими `script`-строками. Перед каждым `DELETE` дашборд заново проверяет desired-state; если hash снова стал нужен ноде, GC отменяется.

Все remote-шаги инициирует дашборд (как и всегда), идут поверх mTLS и считаются best-effort reconciliation поверх уже изменённой локальной БД. Удаление идемпотентно: если hash на конкретной ноде уже отсутствует — агент отвечает успехом без ошибки.

## Детальный дизайн

### `POST /v1/scripts/upload`

|Поле|Тип|Описание|
|---|---|---|
|`script_source`|string|полный текст скрипта|

Ограничение размера `script_source` в v1 — `max_script_upload_bytes = 1 MiB`; запрос с превышением отклоняется до полного чтения тела (`413 Payload Too Large`). На чтение самого тела запроса выставлен таймаут (защита от slowloris).

Запись на диск — атомарная: сначала во временный файл рядом с целевой директорией, затем `fsync`, затем `rename()` в финальное имя (= хэш). Целевая директория доступна на запись только агенту (см. "Ограничения").

Хэш содержимого (SHA-256) считается агентом от фактически полученных байт — единственный источник истины. Любое поле `hash` в запросе игнорируется. Возвращает:

```json
{"hash": "a3f9e2..."}
```

Если скрипт с таким хэшем уже есть на диске — ничего заново не пишем, просто подтверждаем (upload идемпотентен по построению).

### `POST /v1/scripts/execute`

|Поле|Тип|Когда обязательно|
|---|---|---|
|`hash`|string|всегда|
|`request_id`|string (UUID)|всегда — для идемпотентности, см. ниже|
|`args`|string[]|опционально; максимум 64 элемента, 16 KiB на элемент, 64 KiB суммарно|
|`env`|dict[string, string]|опционально; максимум 64 ключа, 128 bytes на ключ, 16 KiB на значение, 64 KiB суммарно|
|`timeout_seconds`|int|опционально; default 60s, максимум 600s|

Response (при успешном нахождении скрипта):

|Поле|Тип|Описание|
|---|---|---|
|`exit_code`|int \| null|null, если процесс убит по таймауту|
|`stdout`|string|машинный результат; превышение лимита = failed-result, не усечение|
|`stderr`|string|диагностика; при превышении лимита обрезается с `stderr_truncated = true`|
|`duration_ms`|int|фактическое время выполнения|
|`timed_out`|bool|true, если убит по таймауту|
|`error`|string \| null|заполняется, если скрипт не успел запуститься по причине, отличной от отсутствия скрипта|

При отсутствии скрипта с указанным хэшем — `404`, без попытки исполнения.

**Идемпотентность через `request_id`.** Ключ кэша — не один `request_id`, а пара `(request_id, hash)` (либо хэш всего тела запроса целиком — эквивалентный по строгости вариант). Если `request_id` совпал с ранее виденным, а остальные параметры запроса (в первую очередь `hash`) отличаются — это ошибка клиента: агент отвечает `409 Conflict`, не пытаясь угадать, какой из двух запросов «правильный». Это закрывает риск случайного переиспользования старого UUID с другими параметрами. Если же запрос полностью совпадает с уже виденным — агент не запускает скрипт повторно, а возвращает сохранённый ответ.

Гарантия идемпотентности действует **только в пределах жизни процесса агента** — кэш `request_id → response` хранится в памяти, не персистится на диск. Если агент перезапустился между выполнением скрипта и повторным запросом дашборда — кэш потерян, и повторный запрос приведёт к повторному запуску. Это сознательно принятое упрощение для v1, а не абсолютная гарантия; явно зафиксировано здесь, чтобы не создавать ложного впечатления у будущего читателя контракта.

Кэш — не неограниченный: `request_id_cache_ttl_seconds = 3600`, `request_id_cache_max_entries = 1024`. Этого достаточно для сетевых retry, scheduler retry и pipeline retry одного логического запуска; более старые записи вытесняются и не дают абсолютной гарантии идемпотентности.

**Гонка execute ↔ delete устранена архитектурно, не поведенческим правилом.** Вместо `stat()`/проверки существования файла с последующим `exec(path)`, агент открывает файл дескриптором (`open()`) и исполняет его через `fexecve(fd, ...)`, а не через путь. Это значит: к моменту запуска файл уже открыт агентом, и параллельный `DELETE` (который лишь убирает запись в директории через `unlink`, не трогая уже открытый inode) не может вызвать `ENOENT` или любую другую гонку — открытый файловый дескриптор продолжает быть валиден и исполним независимо от того, что происходит с directory entry. На платформах, где `fexecve` недоступен, остаётся fallback на `exec(path)` с пониманием, что узкое окно гонки в этом случае возможно (но это уже исключение, а не основной путь).

### `DELETE /v1/scripts/{hash}`

Удаляет файл скрипта по хэшу из пути. Идемпотентен: отсутствие файла с указанным хэшем — это успех (`204 No Content`), не ошибка. Не требует тела запроса. Благодаря `fexecve` (см. выше) не влияет на уже запущенные процессы вообще, без каких-либо оговорок про окна гонки. Дашборд вызывает этот endpoint только после hash-level проверки desired-state: hash удаляется с ноды, только если больше не нужен ей ни через один активный `script`.

### `GET /v1/info`

Не требует тела. Возвращает версию агента и поддерживаемые возможности:

```json
{
  "agent_version": "...",
  "api_version": 1,
  "supported_features": ["script_upload", "script_execute", "script_delete", "admin_unpair", "admin_uninstall"],
  "limits": {
    "max_script_upload_bytes": 1048576,
    "max_execute_body_bytes": 131072,
    "max_stdout_bytes": 262144,
    "max_stderr_bytes": 262144,
    "default_timeout_seconds": 60,
    "max_timeout_seconds": 600
  }
}
```

### Лимиты агента v1

Лимиты — часть контракта агента, а не рекомендательные настройки. Они защищают VPS от случайного DoS пользовательскими скриптами и удерживают модель v1 в рамках синхронного execute, а не общей job-системы.

|Параметр|Значение v1|Поведение при превышении|
|---|---:|---|
|`max_script_upload_bytes`|1 MiB|`413 Payload Too Large`|
|`max_agent_script_storage_bytes`|64 MiB|новые upload отклоняются до GC/cleanup|
|`max_execute_body_bytes`|128 KiB|`413 Payload Too Large`|
|`max_args_count`|64|`422 Unprocessable Entity`|
|`max_single_arg_bytes`|16 KiB|`422 Unprocessable Entity`|
|`max_args_total_bytes`|64 KiB|`422 Unprocessable Entity`|
|`max_env_count`|64|`422 Unprocessable Entity`|
|`max_env_key_bytes`|128 bytes|`422 Unprocessable Entity`|
|`max_single_env_value_bytes`|16 KiB|`422 Unprocessable Entity`|
|`max_env_total_bytes`|64 KiB|`422 Unprocessable Entity`|
|`max_stdout_bytes`|256 KiB|failed-result, `error_class = stdout_limit_exceeded`|
|`max_stderr_bytes`|256 KiB|stderr обрезается, `stderr_truncated = true`|
|`default_timeout_seconds`|60|применяется, если request не указал timeout|
|`max_timeout_seconds`|600|`422 Unprocessable Entity`|
|`speedtest_timeout_seconds`|180|используется системным speedtest-скриптом|
|`shutdown_grace_seconds`|5|после SIGTERM → SIGKILL|
|`max_concurrent_executions_global`|2|`429 Too Many Requests`|
|`max_concurrent_executions_per_hash`|1|`429 Too Many Requests`|
|`request_id_cache_ttl_seconds`|3600|старые id вытесняются|
|`request_id_cache_max_entries`|1024|LRU/TTL-вытеснение|
|`max_processes_per_run`|32|запуск ограничивается RLIMIT_NPROC|
|`max_open_files_per_run`|64|запуск ограничивается RLIMIT_NOFILE|
|`max_memory_bytes_per_run`|256 MiB|запуск ограничивается RLIMIT_AS/cgroup/systemd-run|
|`workdir_quota_bytes_per_run`|64 MiB|запуск завершается failed-result при превышении|

Для pipeline превышение `max_stdout_bytes` всегда считается ошибкой шага: следующий шаг не может безопасно использовать неполный JSON как источник аргументов. `stderr` при превышении обрезается, потому что это диагностический поток; сам факт обрезки возвращается в response.

### Административные endpoint'ы удаления ноды

Административные endpoint'ы используются не для исполнения пользовательских скриптов, а для завершения жизненного цикла ноды в системе. В v1 имена и контракты фиксируются явно:

```text
POST /v1/admin/unpair
POST /v1/admin/uninstall
```

Обе операции доступны только поверх уже установленного mTLS и инициируются дашбордом. Если нода offline, дашборд не может выполнить ни `unpair`, ни полную зачистку; в этом случае возможно только локальное удаление записи ноды из БД дашборда.

#### `POST /v1/admin/unpair`

Разрывает доверие агента к текущему дашборду. Агент удаляет trust anchor/CA дашборда, свой сертификат, private key и локальное pairing-состояние. Штатные `/v1/scripts/*` endpoint'ы после этого больше не принимают команды от старого дашборда. Файлы агента, хранилище скриптов и workdir могут остаться на VPS.

Минимальный успешный ответ:

```json
{
  "status": "ok",
  "agent_state": "unpaired"
}
```

Повторный `unpair` после уже выполненного `unpair` идемпотентен на уровне смысла: агент либо возвращает `200`/`agent_state = unpaired`, если endpoint ещё доступен, либо штатный mTLS уже невозможен, и дашборд трактует это как отсутствие активного pairing. Новый onboarding не должен переиспользовать старый сертификат; он проходит полный bootstrap заново.

#### `POST /v1/admin/uninstall`

Полная зачистка включает `unpair`, но дополнительно удаляет компоненты web-xray-dashboard с VPS: systemd unit, файлы агента, конфигурацию, mTLS-инфраструктуру, хранилище скриптов, workdir и служебные временные файлы.

Минимальный успешный ответ до self-uninstall:

```json
{
  "status": "ok",
  "agent_state": "deinstalling"
}
```

После отправки ответа агент может запустить helper-процесс или `systemd-run` задачу, которая завершит сервис и удалит файлы. Дашборд считает полную зачистку подтверждённой только если endpoint вернул успешный ответ до разрыва соединения. Если соединение оборвалось раньше подтверждения, node не удаляется автоматически: пользователь может повторить uninstall или выбрать локальное удаление.

### Механика запуска процесса

- **Интерпретатор.** Агент не выбирает интерпретатор самостоятельно и не вызывает скрипт через shell. Файл исполняется напрямую через `fexecve` (см. выше), скрипт обязан содержать корректный shebang (`#!/usr/bin/env bash`, `#!/usr/bin/env python3` и т.п.).
- **Рабочая директория.** Не совпадает с директорией хранения хэшей. Используется предсказуемый, стандартизированный путь вида `/var/lib/webxray-agent/work/<uuid>/`, создаётся перед каждым запуском и удаляется после завершения (успешного или нет) в `finally`-блоке самого агента. Дополнительно, при старте агента выполняется резервная очистка (sweep) каталога `work/` от осиротевших директорий, оставшихся от случаев, когда агент аварийно завершился посреди выполнения скрипта и не успел убрать за собой. На один запуск действует `workdir_quota_bytes_per_run = 64 MiB`; превышение завершает запуск failed-result и затем очищается обычным finally/sweep-механизмом.
- **Окружение процесса.** По умолчанию не наследуется от процесса агента целиком. Базовый allowlist: `PATH`, `LANG`, `LC_ALL`, `HOME`, `TMPDIR`, плюс то, что передано в `env` запроса.
- **Пользователь процесса.** Скрипты никогда не исполняются под root — отдельный непривилегированный системный пользователь (например `webxray-agent`).
- **Resource limits.** На каждый запуск применяются ограничения: `max_processes_per_run = 32`, `max_open_files_per_run = 64`, `max_memory_bytes_per_run = 256 MiB`, `workdir_quota_bytes_per_run = 64 MiB`. Конкретная техника — `setrlimit`, cgroup или `systemd-run` — деталь реализации агента, но сами лимиты являются частью v1-контракта.
- **Лимит одновременных выполнений — обязательная часть архитектуры, не опциональная настройка.** В v1: `max_concurrent_executions_global = 2`, `max_concurrent_executions_per_hash = 1`; запрос execute сверх лимита получает `429 Too Many Requests`, а не встаёт в очередь молча. Без этого ограничения серия execute-запросов способна положить ноду. `speedtest` дополнительно сериализован сам с собой и использует `speedtest_timeout_seconds = 180`, так как два параллельных теста испортят оба результата.
- **Завершение по таймауту — явная последовательность, не мгновенный kill.** `default_timeout_seconds = 60`, `max_timeout_seconds = 600`. По истечении `timeout_seconds` процессу (всей process group, через `os.killpg`) сначала отправляется `SIGTERM` и даётся `shutdown_grace_seconds = 5` на самостоятельное завершение; если процесс всё ещё жив — отправляется `SIGKILL`. Это лучше мгновенного `SIGKILL`, который не даёт скрипту шанса прибрать за собой.
- **Чтение stdout/stderr.** Оба потока читаются конкурентно (например через `asyncio.gather`), не последовательно — иначе возможен deadlock. Лимиты раздельные: `max_stdout_bytes = 256 KiB`, `max_stderr_bytes = 256 KiB`. Превышение stdout завершает запуск failed-result (`error_class = stdout_limit_exceeded`), потому что stdout — машинный результат; превышение stderr приводит к обрезке stderr с `stderr_truncated = true`.
- Для метрик-скриптов важен принцип "stdout — только финальный машиночитаемый результат" (предпочтительно JSON в одну строку): прогресс-бары и диагностика должны идти в stderr, а не в stdout. Это ответственность самого скрипта, агент тут ничего специально не парсит.
- **Будущая изоляция (v2).** Вне охвата первой реализации: исполнение через дополнительный слой namespace-изоляции (bubblewrap/nsjail/systemd-run) для более строгой защиты, чем uid + ulimit.

### Реестр скриптов — на стороне дашборда, не агента

Дашборд хранит у себя единую таблицу скриптов вида `имя → текущий хэш`. Никакого отдельного понятия "системный скрипт" нет — метрик-скрипты ничем не отличаются от любых других в этой таблице, кроме того, что их строки создаются программно, автоматически, при инициализации дашборда, а не руками через UI.

Помимо имени и текущего хэша, таблица обязана хранить и сам исходный текст скрипта (поле вида `content`). Без него дашборд не может выполнить `upload` ни на одну ноду — `script_source` в теле запроса берётся именно отсюда, а хэш необратим, восстановить текст из него нельзя. `current_hash` — кэш значения, посчитанного от `content` (тем же SHA-256, что и на агенте); агент остаётся единственным источником истины на момент фактической заливки, но для внутренней бухгалтерии дашборд обязан уметь посчитать его сам, не дожидаясь round-trip к ноде.

Имя скрипта **уникально глобально** по всей таблице (constraint на уровне БД). Попытка создать скрипт с именем, которое уже занято, отклоняется с ошибкой, а не тихо переименовывается.

Когда дашборду нужно запустить скрипт по имени — неважно, пришёл запуск из ручной кнопки UI или из слоя триггеров (`schedule`/`on_startup`), — он смотрит в этой таблице актуальный хэш и дальше идёт по схеме execute → (при 404 на ноде) upload → execute. Каждый фактический вызов `execute` получает `request_id`: новое срабатывание/ручной запуск — новый UUID, retry того же срабатывания — тот же UUID. Если код скрипта поменялся — выполняется флоу обновления (см. выше): строка `script` получает новый `current_hash`, online-ноды получают proactive upload нового hash, а старый hash удаляется через `node_hash_gc` только там, где он больше не входит в desired-state.

### Физическая зачистка hash на нодах

Агент не знает, почему дашборд удаляет hash: скрипт удалён из UI, обновлён, отвязан от папки или перестал быть нужен через `node_script`. Для агента это всегда один и тот же idempotent `DELETE /v1/scripts/{hash}`.

Решение о том, можно ли удалять hash, принимает только дашборд. Правило:

```text
hash можно удалить с node только если hash не входит в desired_hashes(node)
```

`desired_hashes(node)` вычисляется по текущей БД дашборда: все `script.current_hash`, которые доступны ноде через активные строки `node_script`. Если один и тот же content hash используется несколькими именами скриптов, удаление одного `script` не удаляет hash с ноды, пока другой `script` с тем же hash остаётся desired для этой node.

Когда изменение БД уменьшает desired set ноды — удаление `script`, обновление `current_hash`, отвязка `node_script`, удаление `folder_script`, удаление ноды из папки или удаление папки с очисткой связей — код приложения в той же операции создаёт `node_hash_gc(node_id, hash)` для тех hash, которые стали потенциальным мусором. Для online-ноды GC выполняется сразу; для offline-ноды остаётся pending до следующего успешного контакта. Перед выполнением pending GC дашборд заново пересчитывает desired-state: если hash снова стал нужен, GC отменяется.

При удалении самой offline-ноды pending cleanup не создаётся: запись `node` исчезает из БД, и у дашборда больше нет активной сущности, к которой можно привязать отложенную remote-операцию. Это отличается от удаления скрипта: при удалении скрипта offline-нода остаётся активной, поэтому pending GC допустим.

### Заливка метрик-скриптов на ноду

Метрик-скрипты (xray_status, detect_stack, speedtest) — это обычные строки в таблице скриптов на дашборде, просто созданные программно при инициализации дашборда. Никакого особого статуса у них нет. На ноду они попадают тем же `upload`-механизмом, что и любой другой скрипт — разница только в том, что заливку инициирует код дашборда автоматически, без участия пользователя. Когда именно в процессе добавления ноды это происходит — описано в отдельной заметке про онбординг.

## Открытые вопросы

1. ~~**Резервная очистка осиротевших скриптов (хранилище по хэшам).**~~ **Закрыто для основного v1-пути:** физическая зачистка выполняется через `node_hash_gc`, который создаётся при удалении/обновлении скрипта или удалении связи. Лёгкий TTL-sweep на агенте для аварийных краёв (`upload` прошёл, а дашборд умер до записи desired-state) можно оставить как будущую защиту, но не как основной механизм.
2. ~~**Синхронизация обновлений с нодами, которые были недоступны в момент обновления.**~~ **Закрыто для v1:** новая версия доставляется лениво через обычный `execute → 404 → upload → execute`, а старые hash удаляются через pending `node_hash_gc` при следующем успешном контакте с активной нодой.
3. ~~**Квота на размер рабочей директории исполнения.**~~ **Закрыто для v1:** `workdir_quota_bytes_per_run = 64 MiB`; превышение завершает запуск failed-result, рабочая директория очищается обычным finally/sweep-механизмом.
4. **Стоимость create/cleanup рабочей директории на каждый запуск при высокой частоте вызовов.** Если агент исполняет десятки коротких скриптов в секунду, постоянные mkdir/chmod/удаление дерева на каждый вызов могут стать заметными по нагрузке. Возможный компромисс на будущее — пул переиспользуемых рабочих директорий вместо create-per-call, но это оптимизация, а не блокер для первой реализации.
5. ~~**Конкретные значения лимитов.**~~ **Закрыто для v1:** значения зафиксированы в разделе «Лимиты агента v1»: upload 1 MiB, storage 64 MiB, execute body 128 KiB, args/env 64 KiB, stdout/stderr 256 KiB, timeout 60/600s, concurrency 2 global / 1 per hash, request cache 3600s/1024 entries, process limits 32 processes / 64 files / 256 MiB memory / 64 MiB workdir.
6. ~~**Путь и метод bootstrap-эндпоинта (см. «Обзор» выше).**~~ **Закрыто в mTLS-заметке:** bootstrap API вынесен в `/bootstrap/v1/*` (`status`, `csr`, `certificate`) и не пересекается со штатным `/v1/scripts/*` API.
