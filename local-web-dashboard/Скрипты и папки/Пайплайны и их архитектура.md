## Контекст и охват

CORE.md заявляет свободу для продвинутых пользователей: возможность строить собственные пайплайны между нодами. Эта заметка вводит конкретный механизм такой связки — **pipeline** как сохраняемую сущность, состоящую из упорядоченных шагов, где каждый шаг — это исполнение одного скрипта на одной ноде, а вывод (`stdout`) одного шага может быть использован как аргумент другого шага, в том числе не следующего сразу за ним.

Модуль целиком живёт на стороне дашборда и не требует никаких изменений в контракте агента. Агент по-прежнему ничего не знает ни про какие пайплайны — он видит обычную последовательность вызовов `POST /v1/scripts/execute`, неотличимую от ручного запуска скрипта пользователем. Это прямое следствие ограничения из CORE.md: соединение всегда дашборд → агент, агент не может ни инициировать связь с другим агентом, ни вообще знать об их существовании. Оркестрация — исключительно обязанность backend'а дашборда, который последовательно дёргает `execute` на нужных нодах, ждёт синхронный ответ (модель исполнения синхронная, см. Принцип_работы_агента_ноды.md — стриминга и job+polling в v1 нет) и передаёт данные дальше.

## Ограничения

- **Pipeline — сохраняемая, переиспользуемая сущность**, аналогично `script`: создаётся один раз через UI, запускается многократно. Ad-hoc сборка «на лету, без сохранения» не поддерживается.
- **V1 — только линейная цепочка шагов**, без ветвления и merge. Явное допущение (не архитектурный тупик, см. ниже): модель ссылок на аргументы спроектирована так, что шаг N может ссылаться на output любого шага < N, а не только N−1. Это уже даёт часть выразительности DAG (шагу не обязательно нужен именно непосредственный предшественник) при линейном порядке исполнения — и не потребует смены схемы `pipeline_step`/`pipeline_step_arg`, если ветвление появится позже. Требует лишь появления понятия «предок» вместо «позиция − 1», что за пределами охвата этой заметки.
- **Передача данных между шагами — только через `args`, не через `env`.** Скрипт обязан отдавать результат в виде JSON-объекта в `stdout` (та же конвенция, что уже принята для метрик-скриптов в Принцип_работы_агента_ноды.md: «stdout — только финальный машиночитаемый результат», диагностика — в `stderr`). Пользователь при конструировании pipeline в UI явным образом мапит конкретные поля этого JSON-объекта на конкретные позиции `args[]` последующих шагов. Статические (не производные от предыдущего вывода) аргументы задаются как литералы на той же позиции.
- **`env` в v1 не участвует в передаче данных между шагами.** Каждый шаг может иметь свой статический `env` (литеральный, как в обычном ручном запуске скрипта), но не может получить значение из вывода другого шага. Расширение того же механизма маппинга на `env` — открытый вопрос ниже, не блокер для первой реализации.
- **`pipeline_step` обязан материализовать ручную связь в `node_script`.** Если бы pipeline исполнял `execute` на связке (node, script) в обход `node_script`, нода, задействованная только через pipeline, выпала бы из множества рассылки при обновлении хэша скрипта (флоу обновления в Принцип_работы_агента_ноды.md явно опирается на `node_script` как единственный источник этого множества). Материализация реализована триггером, а не кодом приложения — по тому же принципу, что и фан-аут папок в Архитектура_папок_и_материализация_связей.md: партиальное состояние «шаг pipeline создан, а нода не в множестве рассылки» должно быть архитектурно невозможным, а не просто маловероятным.
- **Обратная связь не выполняется.** Удаление `pipeline_step` не удаляет соответствующую строку `node_script`, даже если она была создана этим шагом и больше никем не используется. Асимметрично по отношению к папочному фан-ауту (там revoke — часть той же триггерной пары), и это осознанное упрощение: `node_script`, оставшаяся после удаления шага, не создаёт вреда (скрипт просто продолжит числиться доступным на ноде), тогда как автоматический revoke потребовал бы проверки, не ссылается ли на ту же связку ещё какой-то другой шаг/папка/ручная привязка — довольно затратно ради устранения задолженности, которая безвредна. Зафиксировано как открытый вопрос ниже на случай, если это решение придётся пересмотреть.
- **Ошибка на любом шаге останавливает весь прогон pipeline.** Ни retry, ни rollback предыдущих шагов не предусмотрены — скрипты потенциально имеют побочные эффекты на реальных VPN-нодах, и генерического способа их откатить не существует (в дашборде в принципе нет механизма rollback ни для одной другой операции). Условия остановки: `exit_code != 0`, `timed_out == true`, транспортная ошибка (нода недоступна), или невозможность распарсить `stdout` шага как JSON / отсутствие поля, на которое настроен маппинг для следующего шага.
- **Идемпотентность на уровне отдельного шага полностью переиспользует существующий контракт `execute`** — дашборд генерирует `request_id` (UUID) на пару `(run_id, step_id)` и передаёт его как обычно. Отдельного, специфичного для pipeline механизма идемпотентности не вводится — это тот же принцип «никакого особого статуса», что уже применён к метрик-скриптам.
- **Определение pipeline (`pipeline_step`) и факт его исполнения (`pipeline_run`) — разные сущности**, по аналогии с уже принятым в проекте разделением «намерение vs факт» (`node_script` vs физическое присутствие скрипта на ноде). Между запусками одного и того же pipeline его шаги могут редактироваться; `pipeline_run` обязан фиксировать снимок того, что реально было исполнено (resolved-аргументы, node_id, script_id на момент запуска), а не просто ссылаться на текущее состояние `pipeline_step`, которое к моменту просмотра истории может уже отличаться.
- **История pipeline — техническая часть механизма, а не требование общей истории всех script-run.** `pipeline_run_step.stdout` нужен не только для UI, но и для разрешения `step_output`-аргументов следующих шагов в рамках того же `run_id`. Из этого не следует, что v1 обязан вводить общую таблицу `script_run` для manual/schedule запусков.
- **Pipeline наследует лимиты агента.** Особенно важен `max_stdout_bytes = 256 KiB`: если stdout шага превышает лимит или не является JSON-объектом там, где на него есть ссылка, текущий шаг считается failed, а последующие шаги — skipped.
- **Удаление ноды удаляет будущие pipeline-шаги, но не должно уничтожать историю уже выполненных прогонов.** `pipeline_step.node_id` остаётся FK с `ON DELETE CASCADE`: если нода удалена из дашборда, будущие определения pipeline, ссылавшиеся на неё, теряют соответствующие шаги. `pipeline_run_step` хранит денормализованный снимок `node_id`/`script_id` и результата исполнения, поэтому история прогона остаётся читаемой даже после удаления ноды. Полный lifecycle удаления ноды (`unpair`, полная зачистка VPS, offline-сценарий) описан в Удаление_ноды_из_дашборда.md.

## Обзор

### Таблицы

- **`pipeline`** — `id`, `name` (уникально глобально среди активных pipeline), `archived` (`0/1`, soft-delete), `created_at`. В v1 удаление pipeline с историей = `archived = 1`, не физический `DELETE`.
- **`pipeline_step`** — `id`, `pipeline_id` (FK, CASCADE), `position` (int, порядок в цепочке, уникален в пределах `pipeline_id`), `node_id` (FK → `node`, CASCADE), `script_id` (FK → `script`, CASCADE), `timeout_seconds` (nullable override дефолта агента).
- **`pipeline_step_arg`** — по одной строке на каждую позицию в итоговом `args[]` шага: `id`, `step_id` (FK → `pipeline_step`, CASCADE), `arg_position` (int, позиция в массиве `args`, уникальна в пределах `step_id`), `source_type` (`static` | `step_output`), `static_value` (string, заполнено при `source_type = static`), `source_step_id` (nullable FK → `pipeline_step`, заполнено при `source_type = step_output`; обязан ссылаться на шаг с меньшим `position` в том же pipeline — проверяется в коде приложения, не выражается FK-constraint'ом), `json_field` (string, имя поля в JSON-выводе `source_step_id`, заполнено при `source_type = step_output`).
- **`pipeline_run`** — `id`, `pipeline_id` (FK, RESTRICT — история прогонов не должна теряться при удалении определения pipeline, см. открытые вопросы), `started_at`, `finished_at` (nullable, пока не завершён), `status` (`running` | `succeeded` | `failed`).
- **`pipeline_run_step`** — снимок факта исполнения одного шага в рамках конкретного прогона: `id`, `run_id` (FK → `pipeline_run`, CASCADE), `step_id` (FK → `pipeline_step`, SET NULL — прогон должен остаться читаемым даже если шаг потом удалили из определения), `node_id`, `script_id` (оба — денормализованный снимок на момент запуска, не FK на изменяемое состояние), `request_id` (UUID, передан агенту), `resolved_args` (JSON-массив — реально подставленные значения, после разрешения ссылок на предыдущие шаги), `stdout`, `stderr`, `exit_code`, `timed_out` (bool), `status` (`succeeded` | `failed` | `skipped`), `started_at`, `finished_at`.

### Флоу исполнения pipeline

1. Дашборд создаёт `pipeline_run` со статусом `running`.
2. Для каждого `pipeline_step` по возрастанию `position`:
    - Разрешаются все `pipeline_step_arg` этого шага в порядковый `args[]`: `static` — берётся `static_value` как есть; `step_output` — берётся `pipeline_run_step.stdout` соответствующего `source_step_id` **в рамках этого же `run_id`**, парсится как JSON, извлекается `json_field`.
    - Если парсинг JSON или извлечение поля не удались — текущий шаг и все последующие помечаются `skipped`, `pipeline_run.status = failed`, прогон останавливается.
    - Иначе выполняется `execute` на агенте `node_id` с разрешёнными `args`, новым `request_id`, скриптом `script_id` (по обычной схеме `execute → 404 → upload → execute`, без изменений относительно ручного запуска).
    - Результат (включая `resolved_args`) сохраняется в `pipeline_run_step`.
    - Если `exit_code != 0` или `timed_out == true` или транспортная ошибка — шаг помечается `failed`, все последующие — `skipped`, `pipeline_run.status = failed`, прогон останавливается.
3. Если все шаги завершились успешно — `pipeline_run.status = succeeded`.

### Удаление pipeline с историей

В v1 pipeline не удаляется физически, если у него есть история запусков. Пользовательская операция “удалить pipeline” реализуется как soft-delete:

```sql
UPDATE pipeline SET archived = 1 WHERE id = :pipeline_id;
```

`pipeline_run` и `pipeline_run_step` остаются читаемыми, потому что они являются снимком факта исполнения. Archived pipeline не показывается в активном списке, не может быть запущен вручную и не участвует в будущих trigger/integration-сценариях. Hard delete допустим только как будущая destructive maintenance-операция, не как штатный v1 UI-flow.


## Детальный дизайн

```sql
CREATE TABLE pipeline (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  archived INTEGER NOT NULL DEFAULT 0 CHECK (archived IN (0, 1)),
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE UNIQUE INDEX ux_pipeline_active_name ON pipeline(name) WHERE archived = 0;

CREATE TABLE pipeline_step (
  id INTEGER PRIMARY KEY,
  pipeline_id INTEGER NOT NULL REFERENCES pipeline(id) ON DELETE CASCADE,
  position INTEGER NOT NULL,
  node_id INTEGER NOT NULL REFERENCES node(id) ON DELETE CASCADE,
  script_id INTEGER NOT NULL REFERENCES script(id) ON DELETE CASCADE,
  timeout_seconds INTEGER,
  UNIQUE (pipeline_id, position)
);

CREATE TABLE pipeline_step_arg (
  id INTEGER PRIMARY KEY,
  step_id INTEGER NOT NULL REFERENCES pipeline_step(id) ON DELETE CASCADE,
  arg_position INTEGER NOT NULL,
  source_type TEXT NOT NULL CHECK (source_type IN ('static', 'step_output')),
  static_value TEXT,
  source_step_id INTEGER REFERENCES pipeline_step(id) ON DELETE CASCADE,
  json_field TEXT,
  UNIQUE (step_id, arg_position),
  CHECK (
    (source_type = 'static' AND static_value IS NOT NULL AND source_step_id IS NULL AND json_field IS NULL)
    OR
    (source_type = 'step_output' AND source_step_id IS NOT NULL AND json_field IS NOT NULL AND static_value IS NULL)
  )
);

CREATE TABLE pipeline_run (
  id INTEGER PRIMARY KEY,
  pipeline_id INTEGER NOT NULL REFERENCES pipeline(id) ON DELETE RESTRICT,
  started_at TEXT NOT NULL DEFAULT (datetime('now')),
  finished_at TEXT,
  status TEXT NOT NULL CHECK (status IN ('running', 'succeeded', 'failed'))
);

CREATE TABLE pipeline_run_step (
  id INTEGER PRIMARY KEY,
  run_id INTEGER NOT NULL REFERENCES pipeline_run(id) ON DELETE CASCADE,
  step_id INTEGER REFERENCES pipeline_step(id) ON DELETE SET NULL,
  node_id INTEGER NOT NULL,
  script_id INTEGER NOT NULL,
  request_id TEXT NOT NULL,
  resolved_args TEXT NOT NULL, -- JSON-массив
  stdout TEXT,
  stderr TEXT,
  exit_code INTEGER,
  timed_out INTEGER NOT NULL DEFAULT 0,
  status TEXT NOT NULL CHECK (status IN ('succeeded', 'failed', 'skipped')),
  started_at TEXT,
  finished_at TEXT
);

-- Материализация node_script при создании шага pipeline —
-- по аналогии с фан-аутом папок: партиальное состояние
-- "шаг создан, а нода не в множестве рассылки" архитектурно невозможно.
CREATE TRIGGER trg_pipeline_step_materialize_binding AFTER INSERT ON pipeline_step
BEGIN
  INSERT OR IGNORE INTO node_script (node_id, script_id, folder_id)
  VALUES (NEW.node_id, NEW.script_id, NULL);
END;
```

`source_step_id` обязан ссылаться на шаг с `position` меньше, чем у шага-владельца `pipeline_step_arg` — циклы и ссылки вперёд запрещены. FK этого не выражает (SQLite не умеет сравнивать значения из двух разных строк в constraint), проверка — в коде приложения при сохранении pipeline через UI.

## Открытые вопросы

1. **Асимметрия материализации `node_script`.** Создание `pipeline_step` материализует ручную связь в `node_script`, но удаление шага её не отзывает (см. «Ограничения»). Если со временем накопится ощутимое число «осиротевших» ручных связей от удалённых шагов pipeline — стоит ли пересмотреть в сторону явного revoke-триггера, аналогичного `trg_folder_node_revoke`, с проверкой «не используется ли эта связка ещё где-то» перед удалением?
2. ~~**`pipeline_run` при `RESTRICT` на `pipeline_id`.**~~ **Закрыто для v1:** pipeline с историей не удаляется физически. Пользовательская операция удаления = `UPDATE pipeline SET archived = 1`. История запусков остаётся читаемой, archived pipeline не показывается как активный и не запускается.
3. ~~**Оффлайн-нода посреди прогона.**~~ **Закрыто для v1:** прогон останавливается, текущий шаг получает `failed`, последующие — `skipped`, уже выполненные побочные эффекты не откатываются. UI показывает `pipeline_run_step` по каждому шагу; отдельного retry/rollback-механизма pipeline не вводится.
4. **`env`-маппинг между шагами** — расширение того же механизма (`pipeline_step_arg`-подобная таблица, но для ключей `env`) на будущее, вне охвата первой реализации.
5. **Валидация контракта скрипта.** Ничто не мешает пользователю подключить в pipeline скрипт, который не отдаёт JSON в `stdout` вовсе (сломается только в рантайме, на шаге разбора). Стоит ли вводить необязательное поле-подсказку у `script` (например `output_schema`, не enforced) для документирования ожидаемой структуры вывода при построении pipeline в UI, или это избыточно для v1?
6. ~~**Лимиты на размер pipeline.**~~ **Закрыто для v1:** `max_pipeline_steps = 32`; `step_output` может ссылаться только на любой предыдущий шаг того же pipeline, поэтому отдельная глубина ссылок не вводится. Лимиты `args`, `env`, `timeout`, `stdout` и concurrency наследуются от агента.
7. **UI-поведение при удалении ноды, используемой в pipeline.** БД-правило простое (`pipeline_step` удаляется каскадно), но UI должен предупреждать пользователя, что удаление ноды изменит определения pipeline, где она использовалась. История уже выполненных `pipeline_run` при этом должна остаться читаемой.
