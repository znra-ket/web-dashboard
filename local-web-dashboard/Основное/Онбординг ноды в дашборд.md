## Контекст и охват

Эта заметка детализирует процесс добавления новой ноды в дашборд — от ввода пользователем данных для доступа до момента, когда нода полностью готова к штатной работе (управляется по защищённому каналу, метрики собираются). Обратный процесс — удаление ноды, `unpair` агента и, при необходимости, полная зачистка VPS — описан отдельно в Удаление_ноды_из_дашборда.md.

Согласно CORE.md, дашборд находится за NAT и не имеет публичного IP, соединение всегда инициируется только дашбордом (дашборд → агент), и агент авторизует входящие соединения. Процесс добавления ноды должен соблюдать этот инвариант на всех этапах, включая самый первый, где постоянного защищённого канала ещё не существует.

Процесс разбит на три последовательных этапа:

1. Установка агента по SSH
2. Установка mTLS-канала
3. Заливка метрик-скриптов

Каждый этап — необходимое условие для следующего; пропустить или переставить местами их нельзя.

---

## Ограничения

- Доступ по SSH используется **только** на этапе 1, исключительно для первичной установки. Это единственное место во всей системе, где используется root-пароль ноды.
- После завершения этапа 1 SSH-канал закрывается навсегда — ни дашборд, ни агент больше им не пользуются. Дальнейшее общение идёт только через HTTP (этап 2 — без mTLS, этап 3 и вся последующая эксплуатация — поверх mTLS).
- Root-пароль стирается сразу после этапа 1 и нигде не сохраняется.
- На всех этапах после установки агента действует инвариант CORE.md: инициатива соединения принадлежит только дашборду.

## Обзор

### Этап 1 — установка агента (SSH)

Пользователь предоставляет дашборду IP-адрес ноды и root-пароль. Дашборд подключается к ноде по SSH (asyncssh, см. стек в CORE.md) и выполняет первичную установку: Xray, сопутствующие компоненты, сам агент.

После того как агент установлен и запущен, SSH-канал закрывается навсегда, а root-пароль — стирается. Дальнейшее взаимодействие с нодой идёт исключительно через HTTP, напрямую с агентом.

### Этап 2 — установка mTLS-канала

На ноде агент генерирует ключевую пару и CSR. Дашборд подписывает сертификат. В результате канал дашборд↔агент становится защищён mTLS и остаётся таким для всей последующей эксплуатации ноды.

Конкретный механизм обмена CSR и подписанного сертификата — отдельная заметка (важно, что он должен соблюдать инвариант "только дашборд инициирует соединение", несмотря на то что на этом этапе постоянного защищённого канала ещё не существует).

### Этап 3 — заливка метрик-скриптов

Поверх установленного mTLS-канала дашборд заливает на ноду метрик-скрипты (`xray_status`, `detect_stack`, `speedtest`). Используется штатный механизм `upload`, описанный в Принцип_работы_агента_ноды.md — никакого отдельного пути для метрик-скриптов нет, для агента это обычная заливка скрипта по хэшу.

**По завершении физической заливки дашборд создаёт для этой ноды три ручные записи в `node_script`** (`folder_id IS NULL`) — по одной на `xray_status`, `detect_stack`, `speedtest` (см. Архитектура_папок_и_материализация_связей.md, кейс 13). Это не отдельный шаг ради полноты, а необходимое условие сразу двух вещей:

1. **Включение в множество рассылки обновлений.** Без строки в `node_script` нода не попадёт в выборку при будущей смене хэша метрик-скрипта, и апдейт до неё просто не дойдёт молча, без ошибки.
2. **Автозапуск по расписанию.** Метрик-скрипты — канонический пример schedule-триггера («каждые 6 часов», см. Контекст в Триггеры_запуска_скриптов.md), поэтому каждая из трёх записей создаётся **не пустой presence-связью, а со schedule-триггером**: для каждого метрик-скрипта заводится независимая пара `trigger(type='schedule')` + `trigger_schedule(interval_seconds=21600)`, и её `id` проставляется в `node_script.trigger_id`. Это ровно путь T1 (установка триггера на новую ручную связь) из Триггеры_запуска_скриптов.md — метрики проходят тем же механизмом, что и любая пользовательская связь с расписанием, без особого статуса. После этого T12 (срабатывание schedule-триггера) выбирает эти строки и инициирует штатный флоу `execute → (404) → upload → execute`.

Оба действия — заливка и создание связи с триггером — логически часть одного этапа 3: физическая заливка без привязки не считает этап завершённым, равно как и привязка без расписания (метрики, которые лежат на ноде, но никогда не запускаются, не выполняют своей цели).

Цена этого решения (сознательно принятая, прямое следствие принципа «независимые материализованные копии» — Ограничение п.4 в Триггеры_запуска_скриптов.md): интервал метрик — системная константа, но хранится как N независимых копий `trigger_schedule` (по одной на каждую ноду × каждый метрик-скрипт). Сменить «6 часов» на «4 часа» по всему флоту = обновить все эти строки; единого «system-wide interval» knob'а в схеме нет. Это не частный случай метрик — любой fleet-wide policy на расписаниях платит ту же цену, метрики просто первый, кто по нему проходит.

После этого нода считается полностью добавленной и готовой к штатной работе.

### Lifecycle-состояния ноды во время онбординга

Видимость прогресса для пользователя реализуется не только UI-текстами, а явными backend-состояниями строки `node`. Для v1 вводится минимальный enum:

```text
installing_agent          -- SSH-этап выполняет установку агента
bootstrap_pending         -- агент установлен, bootstrap-token передан, CSR ещё не получен
mtls_pairing              -- CSR получен/подписывается/сертификат устанавливается
metrics_uploading         -- mTLS подтверждён, дашборд заливает метрик-скрипты и создаёт node_script+schedule
active                    -- нода полностью готова к штатной работе
failed_install            -- SSH-установка агента не завершилась
failed_bootstrap_timeout  -- bootstrap не завершился до TTL
failed_mtls_pairing       -- сертификат не установлен или mTLS probe не прошёл
failed_metrics_upload     -- mTLS готов, но метрик-скрипты/связи не созданы полностью
unpairing                 -- нода проходит online local delete через unpair
uninstalling              -- нода проходит полную зачистку VPS
deleting_local            -- дашборд удаляет локальную запись node
```

`deleted` не хранится как активный статус: штатно удалённая нода выражается отсутствием строки `node` в БД. Состояние `unpaired` относится к агенту на VPS после `unpair`, а не к активной записи `node` в дашборде.

UI отображает эти статусы человекочитаемо, но не вводит собственную независимую state machine. Если процесс падает в `failed_*`, пользователь может повторить onboarding/repair-flow; автоматическое продолжение bootstrap без повторного SSH в v1 не поддерживается.

## Открытые вопросы

1. ~~**Механизм обмена сертификатами на этапе 2.**~~ **Закрыто:** полностью описано в Установка_mtls_соединения_дашборд_-_агент.md (bootstrap-токен, передаваемый в конце SSH-сессии этапа 1, временный HTTP bootstrap-эндпоинт агента, опрос дашбордом до готовности CSR, подпись и установка сертификата). Эта заметка выше (раздел «Этап 2») намеренно не дублирует механику — только держит инвариант «дашборд всегда инициирует соединение» как контракт, которому обязана следовать реализация из mTLS-заметки.
2. ~~**Обработка сбоя между этапами.**~~ **Закрыто для v1:** raw bootstrap-token хранится только в памяти текущего onboarding-процесса, TTL токена — 15 минут, абсолютное bootstrap-окно — 30 минут. Если дашборд падает или этап 2 не завершается до TTL, onboarding получает `failed_bootstrap_timeout`; автоматическое продолжение без повторного SSH не поддерживается. Пользователь запускает onboarding/repair-flow заново, и SSH-этап перезаписывает bootstrap-token новым.
3. ~~**Видимость прогресса для пользователя.**~~ **Закрыто для v1:** прогресс отображается через backend enum `node.status`: `installing_agent`, `bootstrap_pending`, `mtls_pairing`, `metrics_uploading`, `active`, `failed_*`, `unpairing`, `uninstalling`, `deleting_local`. UI только отображает эти состояния человекочитаемо и не вводит отдельную state machine.
4. ~~**Обратный процесс — удаление ноды из дашборда — нигде не описан.**~~ **Закрыто:** обратный процесс вынесен в Удаление_ноды_из_дашборда.md. Там зафиксированы два режима: локальное удаление из дашборда и полная зачистка VPS. Для online-ноды даже локальное удаление сначала выполняет `unpair`, чтобы агент перестал доверять старому дашборду; для offline-ноды доступно только локальное удаление с предупреждением, что агент, mTLS-файлы и скрипты могли остаться на VPS.
