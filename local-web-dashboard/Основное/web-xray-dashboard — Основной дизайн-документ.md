## Контекст и охват

- Пользователь управляет несколькими VPS-серверами для VPN. Дашборд берёт на себя их первоначальную настройку (включая установку и конфигурацию Xray и сопутствующих компонентов) и дальнейшее централизованное управление, упрощая типовые операции до уровня "одной кнопки". Помимо этого, дашборд даёт свободу продвинутым пользователям: возможность загружать свои скрипты и строить собственные пайплайны между нодами (конкретный механизм группировки нод и скриптов — папки, см. отдельную заметку).
- Проект разрабатывается как продукт для публичного релиза, а не как персональный инструмент.

## Ограничения

- **Никогда не SaaS** — дашборд всегда стоит локально, на компьютере/сервере владельца. Никогда не хостится централизованно для множества клиентов одним инстансом.
- **Один дашборд управляется одним владельцем/командой**, и никакая нода физически не может одновременно управляться двумя разными инсталляциями дашборда (то есть нет конфликта владения — нода либо принадлежит к экосистеме одного дашборда, либо нет). Это инвариант на уровне модели владения, а не просто "не-цель" про мультитенантность — он жёстче.
- **Дашборд находится за NAT и не имеет публичного IP.** Агент не может инициировать соединение к дашборду. Соединение всегда только в одну сторону: дашборд → агент.
- **Агент авторизует входящие соединения** — агент принимает и исполняет команды только от авторизованного дашборда. Несанкционированные запросы отклоняются. Конкретный механизм аутентификации описан в детальном дизайне.
- **mTLS-сертификат агента валиден для дашборда только вместе с активной записью `node`.** Успешная TLS-проверка цепочки сама по себе недостаточна: дашборд дополнительно сопоставляет fingerprint/serial/public key fingerprint сертификата агента с активной записью ноды в локальной БД. Удаление `node` удаляет это прикладное доверие без отдельного CRL.
- **Жизненный цикл ноды симметричен онбордингу.** Нода может быть удалена только из локальной БД дашборда либо полностью зачищена на VPS. Для online-ноды даже простое удаление сначала выполняет `unpair` — агент забывает доверие к старому дашборду. Для offline-ноды доступно только локальное удаление с явным предупреждением, что агент и его mTLS-инфраструктура могли остаться на VPS.
- **Desired-state и физическое состояние разделены.** Локальная БД дашборда хранит намерение (`node_script`, `script.current_hash`, папки, триггеры), а агент хранит физические файлы по content hash. Если изменение БД делает старый hash больше не нужным активной ноде, дашборд ставит hash в очередь `node_hash_gc` и удаляет его с агента best-effort: сразу для online-ноды, позже для offline-ноды при следующем успешном контакте.
- **Лимиты агента — часть архитектуры v1.** Агент не является неограниченной job-системой: upload, execute body, stdout/stderr, timeout, concurrency, request_id-cache, память, число процессов, open files и workdir имеют фиксированные лимиты, описанные в Принцип_работы_агента_ноды.md.

## Обзор

Система состоит из двух сущностей: **Дашборд** и **Агент**.

- **Дашборд** — находится на локальном сервере/компьютере владельца. Состоит из UI (взаимодействие с пользователем) и backend (взаимодействие с агентами).
- **Агент** — находится на каждой ноде (VPS). Принимает команды от backend дашборда и выполняет их непосредственно на ноде, возвращая результат обратно.

Инициатива всегда исходит от дашборда: backend дашборда обращается к агенту с командой, агент выполняет её на ноде и отдаёт ответ.

### Implementation index

Перед началом реализации текущий пакет заметок сводится в `ARCHITECTURE_INDEX.md`. Этот индекс не заменяет тематические документы, а фиксирует текущую v1-карту системы: таблицы, endpoint'ы агента, фоновые процессы, lifecycle-состояния ноды, закрытые решения и намеренно отложенные v2-вопросы.

Границы транзакций и правило «локальный desired-state в SQLite, remote-действия только после commit» зафиксированы отдельно в `Transactional_service_boundaries.md`. Это обязательная часть реализации: SQL-триггеры отвечают только за локальную консистентность, но не вызывают сеть и не запускают операции на агентах.


## Детальный дизайн

### Стэки и технологии

**Frontend:** React 19, TypeScript, Vite, Tailwind CSS v4, Zustand, Axios, Lucide React

**Backend:** Python, FastAPI + Uvicorn, SQLAlchemy (asyncio), Pydantic, APScheduler, asyncssh (только для первичной установки агента на ноду)

**База данных:** SQLite (aiosqlite), один файл — vpn_dashboard.db

**Инфраструктура:** локальный запуск через venv + npm, без Docker

**Агент:** отдельный сервис на каждой ноде, взаимодействие через HTTP
